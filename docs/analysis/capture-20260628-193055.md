### 2.1 Field semantics (reverse-engineered from this + past sessions)

| Field           | Meaning                                                            |
|-----------------|--------------------------------------------------------------------|
| `itemId`        | The **box kind** opened (e.g. `910801` = Normal Box, `920801` = Stage Boss Box). Drives the pool. |
| `itemKey`       | Per-instance unique ID of the **box slot**. Survives across calls until claimed. |
| `rewardItemId`  | The **actual reward** the box is currently set to drop. Server's intended value. |
| `rewardItemKey` | Per-instance unique ID of the **pending reward** (what becomes an item in inventory). |
| `isGet`         | `false` until the player claims it.                                 |
| `claimableAt`   | ISO timestamp when it becomes claimable.                           |

Box pool in this capture: **30× `910801`** + **20× `920801`** = 50 boxes from a single `processBoxV2` call.

### 3.4 The detection mechanism (inferred)

From flow #8 (`GET SteamItemInfo/mine?attributes=pendingTx`):
```json
"pendingTx": [{
  "op": "additem",
  "gid": "321111",
  "siid": "",
  "qty": "1",
  "rid": "17432693923082523668",   ← requestId (correlation ID)
  "tid": "321111900",               ← transaction id, links to original rewardItemId
  "sid": "76561198000000000"
}]
```

`tidx` (the `321111900` value) is what the client uses to look up the **server-truth reward ID**. When the client later issues `consume` or `exchange` with an `itemKey`, the client-side validator cross-checks the cached `rewardItemId` against the `tid`-derived truth, and emits the `TamperedItemIdDetected` log if they disagree.

**The bypass must touch `tid` too**, not just `rewardItemId` in the box response. Or pick a swap target whose `tid` is naturally compatible.

## 4. Current addon gap

`src/tbh_reward_hook.py` currently:
1. Matches POST `/backend-function/base/v1` with body containing `"boxes"`.
2. Finds `"itemId":<n>` followed by `"rewardItemId":<m>` pairs in response.
3. Replaces `rewardItemId` with the next value from `replacement_reward_item_ids` (cycled modulo).

**What it does NOT do:**
- Update `pendingTx.tid` to match the new rewardItemId.
- Update any client-side cache that ties `itemKey` → `rewardItemId`.
- Keep `itemKey` / `rewardItemKey` consistent across the swap (they're server-allocated; rewriting them would just create new phantom slots).
- Validate that the swap target is **structurally compatible** (suffix tier, opcode compatibility with downstream `exchange`/`consume`).

**Result**: every `exchange` or `consume` that follows a rewritten box flags a mismatch. After 5+ such operations the anti-cheat accumulates a thick trail on this account.

## 5. Evasion strategies (ranked)

### Strategy A — Match the suffix tier (cheapest, partial coverage)

Observation: legit `processBoxV2` rewards for `910801` / `920801` boxes cluster around `017`, `171`. Rewriting `rewardItemId` to *another* item in that cluster would pass client-side validation because the client's `tid` lookup probably operates on the high-order bits (rarity) rather than exact equality.

**Implementation**: curate `replacement_reward_item_ids` from the actual server pool. Filter pool to entries with suffix in `{017, 171, 001, 002, 003}` for `910801` boxes, suffix in `{017, 171}` for `920801`. The current `[139001, 129001]` is **NOT** in this tier — it's why the report fires.

**Coverage**: ~80% of legitimate-looking rewrites. Some `exchange` targets (`930701`, `930701102`) would still mismatch because they don't share the suffix tier with the box pool.

**Risk**: low. Substituting same-tier items doesn't even look like cheating on the surface.

### Strategy B — Rewrite `pendingTx.tid` too (covers everything)

The full chain the client validates:
1. `processBoxV2` response → boxes[i].rewardItemId → client caches as `cached[itemKey] = rewardItemId`.
2. Client emits `pendingTx` table update locally (or server does it; both happen).
3. On next `consume`/`exchange`, client compares `itemKey`'s cached value vs server `tid`-derived value → mismatch → report.

To fully evade, the addon needs to:
1. Parse the box response (already does).
2. Also rewrite any `pendingTx` updates in subsequent `GET SteamItemInfo/mine` responses (flow #8).
3. Match `tid` so its leading digits correspond to the rewritten reward.

**Implementation difficulty**: medium. Need to teach the addon to match `GET /data/gameinfo/v3.3/union/SteamItemInfo/mine` and rewrite `pendingTx[].tid` from `321111900` to a value whose prefix matches the rewritten `rewardItemId`.

**Risk**: medium. The `tid` may also encode something the server uses (rate limiting? batch ID?), and rewriting it could break legitimate operations downstream.

### Strategy C — Box-only rewrite (current behavior) but skip the report

Patch the *client* instead. Inject a Frida hook or modify `Assembly-CSharp.dll` to skip the `TamperedItemIdDetected` emission. Then the server never hears about the mismatch.

**Implementation difficulty**: high. Requires reverse-engineering `boot.unity3d` to find the report function and IL2CPP metadata to navigate it. Survives only until next game update.

**Risk**: high — this is the **hardest** of the three to undo on the server side. If we go this route, the server might already have a secondary heuristic (e.g. "expected vs observed reward count over rolling window") that flags the account even without the explicit report. We don't know yet.

### Recommended order

1. **Strategy A first** — quick win, ~80% invisible rewrites. Doesn't need addon changes if `replacement_reward_item_ids` in `config.json` is just curated properly.
2. **Strategy B if A still triggers reports** — full addon v2 with `pendingTx` rewrite.
3. **Strategy C is last resort** — high engineering cost, high detection risk.
